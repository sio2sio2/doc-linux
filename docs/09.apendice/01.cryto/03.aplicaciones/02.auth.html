9.1.3.2. Autenticación¶

Algunos servicios usan para la autenticación de los clientes una técnica de clave pública como alternativa a la típìca autenticación con contraseña. El mecanismos de autenticación, a grandes rasgos, se basa en lo siguiente:

  1. Previamente y de algún modo seguro, el cliente ha comunicado su clave pública al servidor y este la registra para usarla cuando el cliente desee autenticarse.

  2. Cuando el cliente se autentica, indica la clave pública con la que desea identificarse.

  3. Si el servidor tiene esa clave pública almacenada, generará un token aleatorio, lo cifrará con tal clave pública y lo remitirá al cliente.

  4. El cliente obtiene el token descifrándolo con la clave privada y lo devuelve al servidor1 para que este se cerciore de que es el propietario de esa clave.

  5. El servidor, una vez comprueba que el cliente ha sido capaz de leer el token, lo reconoce como legítimo y le da acceso.

A lo largo del manual, hay ejemplos de cómo se configura este tipo de autenticación en diversos servicios:

  • Identificación con clave pública en SSH.

  • Identificación del cliente con certificado en OpenVPN.

  • Configuración de wireguard, que sólo admite la autenticación del cliente mediante clave pública.

La autenticación con clave pública presenta algunos beneficios respecto a la autenticación con contraseña:

  • Inutiliza los ataques de fuerza bruta, ya que dejará de haber contraseñas que adivinar.

  • Puede usarse una misma clave pública para la identificación en distintos servidores. De este modo la contraseña de acceso será la misma en todoss: la clave simétrica con la que se haya cifrado la clave privada.

Notas al pie

1

En realidad, la devolución del token puede ser más sofisticada. Por ejemplo, el cliente SSH descifra el token, lo combina con la clave de sesión, genera un resumen con md5; y este resumen el que remite al servidor. Para una explicación detallada del proceso lea el artículo Understanding the SSH Encryption and Connection Process
